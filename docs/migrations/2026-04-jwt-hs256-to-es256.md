# SoundBridg — Presence Auth Revision (proposal)

**Status:** draft / awaiting review.
**Author:** Claude (Session 4 prep, 2026-04-27).
**Supersedes (if accepted):** PRESENCE_PROTOCOL.md §5 — Path A1 (legacy HS256).
**Will commit only after your explicit approval.**

---

## TL;DR

PRESENCE_PROTOCOL.md §5's Path A1 plan is **no longer executable** on our Supabase project. Recommend **switching to a variant of Path A2** — backend signs with an **ES256 (P-256) asymmetric key** that we generate locally and **import into Supabase as a standby signing key**. Our `JWT_SECRET` (HS256) is retired from the auth path; the backend's signing key becomes a private PEM stored in Render env. Realtime validates incoming JWTs via the project's JWKS (which now contains our public key alongside Supabase's existing ECC key).

This costs one forced re-login event for every active user (same blast radius as a `JWT_SECRET` rotation, which CONSTRAINTS §1 already documents as expected on rotation), and requires three constitutional edits (CONSTRAINTS §1, PRESENCE_PROTOCOL §5 + §11, ROADMAP §3 Week 3). A1 is dead even if we rotate Standby flows; A3 (Supabase-issued tokens) is not supported for generic custom backends and would also reopen CONSTRAINTS §1's "no Supabase Auth" line. **A2 is the only path that ships.**

## 1. Findings (what we know now)

### 1a. From the user's tonight-investigation of the Supabase dashboard

1. Project has been migrated to **ECC P-256** JWT Signing Keys.
   - Current key id: `7F717609-DB07-4D69-A07A-5934EA44FAF4`.
2. Legacy HS256 secret still exists, but is **verify-only**.
   - Dashboard: *"Legacy JWT secret can only be changed by rotating to a standby key and then revoking it. It is used to only verify JSON Web Tokens by Supabase products."*
3. Render `JWT_SECRET` (backend signing key) and Supabase's legacy HS256 secret have **never matched**. Backend custom JWTs have never been Realtime-validatable.
4. Path A1's plan ("set Supabase legacy = `JWT_SECRET`") is blocked because the legacy secret's value is not editable on a migrated project — only rotatable through the Standby flow.

### 1b. From current Supabase docs (verified 2026-04-27, sources in §10)

5. **Private key / shared secret are not extractable from Supabase post-migration.** The signing-keys doc states explicitly: *"Once you've moved to using the JWT signing keys feature extracting of the private key or shared secret from Supabase is not possible."* So we cannot copy Supabase's legacy HS256 value into Render's env to coerce A1 from the other direction either.
6. **Bring-your-own-key is supported.** Supabase accepts an externally-generated keypair as a signing key; the *public* key is registered with the project, the *private* key stays in our backend. The `supabase gen signing-key --algorithm ES256` CLI is the convenience path; manual openssl-then-import works equivalently.
7. **A standby asymmetric key's public key IS published in the project's JWKS discovery endpoint** (`https://<ref>.supabase.co/auth/v1/.well-known/jwks.json`). Realtime fetches JWKS and trusts every published public key for verification, regardless of whether the key is "in use" or "standby" (only "revoked" excludes it). Standby just means "Supabase Auth doesn't *issue* with it" — but Realtime still *verifies* against it.
8. **Multiple signing keys coexist by design.** in-use + standby + previously-used are all trusted for verification simultaneously. The lifecycle states are: standby → in-use → previously-used → revoked, and all transitions are reversible (except permanent deletion).
9. **Generic custom JWT issuers are not in the supported third-party-auth list.** Supabase explicitly supports five providers (Clerk, Firebase, Auth0, Cognito, WorkOS); a generic "we are the issuer, here's our JWKS" registration slot is not documented as a first-class flow. The supported way to make Supabase trust our backend's JWTs is via the signing-keys import (point 6), not via third-party-auth.

## 2. Re-evaluation of A1 / A2 / A3 against current reality

### Path A1 — revert Supabase to HS256-with-our-secret
**Verdict: not viable. Drop from PRESENCE_PROTOCOL.md.**

Two sub-options were considered:

- **A1.i — coerce Render to use Supabase's legacy HS256 value.** Set `JWT_SECRET` on Render = Supabase's existing legacy secret (which we have read-only visibility on via dashboard). Forces re-login for every user. *But:* the legacy secret is on Supabase's deprecation track — once we revoke it (or Supabase eventually deprecates it), Realtime stops trusting tokens signed with it. We'd be building presence on a tombstone.

- **A1.ii — rotate the legacy HS256 secret back to "in use" via Standby flow.** Per docs, a previously-used key can be moved back to standby and rotated. But the secret's *value* doesn't change — Supabase generated it, and we still can't set it equal to our `JWT_SECRET`. So we'd land back in A1.i (coerce Render to match Supabase) with the same deprecation tail risk, plus the additional risk of rotating the in-use key (potential anon/service_role re-issuance, additional moving parts).

In both cases, the symmetric-secret model is on a managed exit ramp. Even if A1 is technically reachable today, we're running counter to the platform's direction for a v1 feature whose payoff is dwarfed by the migration cost. Drop A1.

### Path A2 — backend signs ES256 with a key we own
**Verdict: chosen. Variant of original A2, refined below.**

The original PRESENCE_PROTOCOL.md §11 framing of A2 was "backend signs with an asymmetric key, future work." The platform-current shape is more concrete than that:

- Generate a P-256 keypair locally (`openssl ecparam -name prime256v1 -genkey -noout -out signing.pem; openssl ec -in signing.pem -pubout -out signing.pub.pem`).
- Compute a `kid` (UUID v4 is fine; hash-of-public-key also fine).
- Import the keypair into Supabase as a **standby** signing key. The public key appears in JWKS; the private key (the PEM file) goes into Render env as `JWT_PRIVATE_KEY`.
- Backend swaps `jsonwebtoken` calls from `HS256 + JWT_SECRET` to `ES256 + JWT_PRIVATE_KEY` with the `kid` header set. Auth middleware swaps verify from `HS256 + JWT_SECRET` to `ES256 + JWT_PUBLIC_KEY` (or fetches JWKS — but holding the public key locally avoids a network hop on every request).
- Existing ECC P-256 in-use key stays in use. Anon/service_role keys are unaffected.
- Realtime validates incoming connection JWTs against JWKS, finds our public key, accepts.

**Risk surface:**
- One forced re-login: every active 30-day session is signed under the old HS256 secret. Post-cutover, none verify under the new key. CONSTRAINTS §1 already documents this as the cost of a `JWT_SECRET` rotation; A2 is operationally equivalent.
- The private key now lives in Render env as a multi-line PEM (vs. a single-line secret today). Render supports multi-line env vars natively; no infra change needed.
- One additional dependency on the Supabase project's signing-key configuration. If our standby key were ever inadvertently revoked, every backend-issued token instantly fails verification — same blast radius as a `JWT_SECRET` rotation, but the trigger is on the Supabase dashboard, not on Render. Mitigation: dashboard discipline + the standby key's "previously used" state being reversible (Supabase keeps it trusted until *revoked*, which is the destructive operation; standby/previously-used are both trusted).
- Backend must also begin signing with `sub` and `role: "authenticated"` claims (already planned for A1 per Session 1's commit `b545ac3` — already shipped). No additional payload work.

### Path A3 — Supabase-issued tokens (Auth)
**Verdict: not viable + violates CONSTRAINTS §1.**

- Generic custom JWT issuers are not in the supported third-party-auth list (Clerk/Firebase/Auth0/Cognito/WorkOS only).
- Adopting Supabase Auth as the issuer means migrating users off our bcrypt+custom-JWT model onto Supabase's user table. CONSTRAINTS §1 prohibits this without "a coordinated cross-repo migration plan." That plan does not exist and is not in scope for Phase 2.
- A3 is the right end-state for a *different* product (one where Supabase is the user system); SoundBridg is not that product today.

---

## 3. Recommendation (the chosen path in one sentence)

**Adopt A2. Generate an ES256 (P-256) keypair, import the public half as a Supabase standby signing key, store the private half in Render env, and switch backend JWT mint+verify from HS256 to ES256.** Do this as a single coordinated change with a one-time forced re-login for all users, then unblock Session 4 (mobile presence wiring) on the new key.

## 4. Constitutional changes required

Listed in the order they'd land. Each is a small, surgical edit; no doc rewrites.

### 4a. CONSTRAINTS.md §1 — Auth model

Three lines change. From the current text:

> **Custom HS256 JWT. 30-day expiry. No refresh token.** Do not introduce Supabase Auth. Do not introduce OAuth, magic links, or passkeys without a cross-repo migration plan.
> **`JWT_SECRET` is the one signing key.** No per-client secrets, no asymmetric migration, no co-existing signing schemes. Rotating `JWT_SECRET` invalidates every active session — plan accordingly.
> **Legacy JWT secret is coupled to `JWT_SECRET` under Path A1.** [...]

To:

> **Custom ES256 JWT. 30-day expiry. No refresh token.** Do not introduce Supabase Auth. Do not introduce OAuth, magic links, or passkeys without a cross-repo migration plan. (Algorithm changed from HS256 to ES256 in Week 3 to enable Supabase Realtime auth via the JWKS path; see PRESENCE_AUTH_REVISION for context.)
> **`JWT_PRIVATE_KEY` is the one signing key.** Stored as a P-256 PEM in Render env; the matching public key is registered with the Supabase project as a standby signing key and published in the project's JWKS. Rotating `JWT_PRIVATE_KEY` requires a coordinated rotation: import a new standby keypair on Supabase, swap Render env, retire the old standby. Either step alone breaks auth — plan the rotation as a single operation. Rotation invalidates every active session.
> **Legacy HS256 coupling is retired.** The Path A1 framing (Supabase legacy HS256 secret = `JWT_SECRET`) was made unreachable when the project migrated to ECC signing keys. The current model uses asymmetric verification via JWKS — Supabase trusts our public key without ever holding our private key. CONSTRAINTS §2 ("backend is the single gatekeeper") strengthens, not weakens, under this model: Supabase cannot mint tokens that look like ours; only the holder of `JWT_PRIVATE_KEY` can.

The line *"Bearer transport only"* and the per-client storage rules are unchanged. The 401-clears-session rule is unchanged. The token payload shape now adds `sub` + `role` (already shipped in `b545ac3`) but otherwise unchanged.

### 4b. PRESENCE_PROTOCOL.md §5 — Auth (largest edit)

Rewrite §5 wholesale. New title: **"Auth — Path A2 (asymmetric ES256 via JWKS)."** Outline:

- **Mechanism**: backend mints ES256-signed JWTs with `kid` header matching the standby key's id; Supabase Realtime fetches JWKS, sees our public key, validates connection.
- **Connection flow**: unchanged from current §5 except the "set Supabase legacy secret = `JWT_SECRET`" step is replaced by "import our public key as a Supabase standby signing key (one-time setup)."
- **Why A2 not A1/A3**: replace the existing "Why A1 not A2/A3" justification with a "Why A2 not A1/A3" one. A1 is now in the dead-paths section; A3 is unchanged (not supported, would reopen CONSTRAINTS §1 anyway).

### 4c. PRESENCE_PROTOCOL.md §6 — RLS policies

**No change.** The RLS policies key off `auth.uid()` and `realtime.topic()`, which work identically under A2. The `role: "authenticated"` claim still gates `to authenticated` in the policy.

### 4d. PRESENCE_PROTOCOL.md §11 — Open items

Strike "Path A2 / A3 auth migration" line; A2 is the implemented path now, A3 stays out of scope.

### 4e. ROADMAP.md §3 Week 3

One-time forced re-login for all active sessions. Required to unblock Session 4 (mobile presence wiring).

## 5. Migration plan (NOT to be executed tonight)

Listed for review only. None of these run until you approve A2.

1. **Generate keypair locally** (one developer machine, dev-controlled):
   ```
   openssl ecparam -name prime256v1 -genkey -noout -out signing.pem
   openssl ec -in signing.pem -pubout -out signing.pub.pem
   # kid: any uuidgen output, or `openssl ec -in signing.pem -pubout -outform DER | openssl dgst -sha256 -hex`
   ```
   `signing.pem` never leaves the dev machine until step 5; `signing.pub.pem` is what gets imported to Supabase.

2. **Import the public key into Supabase as a standby signing key.** Either via the dashboard's "import standby key" flow or via the CLI (`supabase gen signing-key --algorithm ES256 --import signing.pub.pem` if that flag exists; otherwise dashboard). Verify it appears in `https://<ref>.supabase.co/auth/v1/.well-known/jwks.json` with our `kid`.

3. **Stage backend changes** on a feature branch (no merge yet):
   - Add `JWT_PRIVATE_KEY` and `JWT_PUBLIC_KEY` envs to `.env.example` (PEM block placeholders + a comment pointing at PRESENCE_AUTH_REVISION).
   - Update `server.js`'s `jwt.sign(..., JWT_SECRET, { algorithm: 'HS256' })` to `jwt.sign(..., JWT_PRIVATE_KEY, { algorithm: 'ES256', keyid: KID })`.
   - Update `authMiddleware`'s `jwt.verify(token, JWT_SECRET)` to `jwt.verify(token, JWT_PUBLIC_KEY, { algorithms: ['ES256'] })`.
   - Backend continues signing `sub` + `role: 'authenticated'` (already in `b545ac3`).
   - Local test: hit `/api/auth/login`, decode the returned token, confirm `alg: 'ES256'` and `kid: <our-kid>`, confirm verify round-trips.

4. **Stage Supabase Realtime end-to-end test.** From a scratch script using `@supabase/supabase-js`, log in via the dev backend (now ES256), call `supabase.realtime.setAuth(jwt)`, subscribe to `presence:user:<id>`, `track()` a payload, confirm the channel accepts the connection. This validates the JWKS path before we touch production.

5. **Production cutover** (single coordinated deploy):
   - Set `JWT_PRIVATE_KEY` and `JWT_PUBLIC_KEY` on Render. Optionally keep `JWT_SECRET` set briefly (for rollback safety — see step 7).
   - Deploy backend with ES256 mint + verify.
   - All existing 30-day HS256 tokens immediately fail verification → users hit 401 → mobile/web/desktop session-clearing fires (existing behavior per CONSTRAINTS §1) → user re-logs in → new token is ES256.
   - Comms (in-app banner / status page note): "You'll need to sign in again — security upgrade."

6. **Verify Realtime works** on the production project: spin up two browser tabs, sign in on both, confirm presence cross-fires.

7. **Retire `JWT_SECRET`** after a 7-day soak. Remove from Render env, remove from `.env.example`, remove the variable from `server.js` (only the constant declaration; all consumers were already swapped in step 3).

8. **Land the doc edits** (CONSTRAINTS §1, PRESENCE_PROTOCOL §5/§11, ROADMAP §3, ARCHITECTURE §5). Delete `docs/PRESENCE_AUTH_REVISION.md` (this file) — the changelog is the git history.

## 6. Risks and rollback

| Risk | Mitigation |
|---|---|
| Forced re-login affects every active user | One-time only. CONSTRAINTS §1 already documents rotation cost. Comms via in-app banner. |
| Standby key revoked by accident in Supabase dashboard | Standby and previously-used are both reversible; only revoked is destructive. Discipline + don't click revoke. |
| Backend deploy ships with new code but old envs (or vice versa) | Single-deploy: env change + code change in same Render deploy. Pre-deploy local test (step 4) catches mismatch. |
| `jsonwebtoken` lib doesn't accept multi-line PEM cleanly | It does; node-jsonwebtoken supports PEM strings directly. Verified pattern. |
| Render env doesn't tolerate multi-line PEM | Render supports multi-line env vars natively. Verified. |
| OTA mobile clients get a 401 storm at cutover | Expected. Mobile already installs a global 401 handler that flushes the session (per ARCHITECTURE §5). User signs in once, done. |
| Need to roll back fast | If JWT_SECRET is kept on Render through cutover (step 5 optional flag), revert deploy → backend signs HS256 again → existing tokens that were re-issued ES256 in the bad window become invalid (small re-login wave). Acceptable. |

## 7. Open questions for your review

These are decisions only you should make; this proposal is not committing to any of them:

1. **Cutover timing.** I recommend running the migration BEFORE Session 4 starts (since Session 4 needs Realtime auth working). Are you OK with the forced re-login event happening this week, or do you want to defer until a quieter window?
2. **`JWT_SECRET` retention period after cutover.** I default to 7 days as the soak-and-rollback window. Shorter (24h) is fine if you're confident; longer (30 days, full token lifetime) is also fine and gives a fuller-coverage rollback window at zero ongoing cost.
3. **Comms.** Should we ship an in-app banner explaining the re-login, or just let the 401 happen silently? Silent is what the current codebase does; banner is more polite but requires a separate small surgery.
4. **Key rotation cadence going forward.** Once on ES256, we own the rotation cadence. I'd suggest annually unless we hit a leak. Do you want a rotation TODO captured somewhere now, or as future work?
5. **Doc-edit packaging.** Ship CONSTRAINTS / PRESENCE_PROTOCOL / ROADMAP / ARCHITECTURE edits as one commit alongside the code change, or split (docs first, code second)? I'd default to single commit because the docs claim the new behavior is in production — separating them creates a window where docs and code disagree.
6. **`docs/PRESENCE_AUTH_REVISION.md` lifecycle.** Delete on landing (my default), or keep as a permanent migration record under `docs/migrations/`?

## 8. What does NOT change

For completeness — the parts of the auth model that stay identical:

- The custom JWT remains the only token any client ever stores (no second token per CONSTRAINTS §1).
- 30-day expiry, no refresh token, bearer transport only (CONSTRAINTS §1).
- Per-client storage (web `localStorage`, desktop `electron-store`, mobile `expo-secure-store`) (CONSTRAINTS §1).
- 401 clears the session (CONSTRAINTS §1, ARCHITECTURE §5).
- bcrypt cost factor 10 for password hashing (CONSTRAINTS §6).
- `sub` + `role: 'authenticated'` claims (already shipped in `b545ac3`).
- The backend remains the single gatekeeper for Postgres + R2 (CONSTRAINTS §2).
- RLS policies on `realtime.messages` (PRESENCE_PROTOCOL §6) — work identically under ES256.

Anyone reading this who's worried about a deeper auth-model rewrite: there isn't one. Algorithm + key shape change only. Everything else stays.

## 9. Why "do nothing" is also worth considering

For honesty: a third option is to defer presence indefinitely.

- **Cost of doing nothing:** the bridg icon's Nearby state remains a static stub forever; Phase 2's "internet-presence demo" loses its only cross-device signal; PRESENCE_PROTOCOL.md becomes a doc that describes a feature we never shipped.
- **Cost of A2:** one weeknight surgery + one forced re-login event + four short doc edits.

A2's cost is lower than do-nothing's product cost. **Recommend ship.**
